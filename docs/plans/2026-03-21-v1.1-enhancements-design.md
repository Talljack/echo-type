## Feature 1: FSRS Spaced Repetition Scheduling

**Goal:** Replace the current heuristic-based review scheduling with the FSRS (Free Spaced Repetition Scheduler) algorithm for scientifically optimized memory retention.

**Current State:**
- `src/lib/daily-plan-progress.ts:92-98` has `nextReviewDelayDays()` — a simple formula based on accuracy + attempt count
- `LearningRecord` has `nextReview?: number` field, no ease factor or state tracking
- Review page at `/review/today` works but uses naive scheduling

### 1.1 Data Model Changes

Add FSRS fields to `LearningRecord` (Dexie schema version 7):

```typescript
interface LearningRecord {
  // ... existing fields ...
  // New FSRS fields
  fsrsCard?: {
    due: number;           // next review timestamp
    stability: number;     // memory stability (days)
    difficulty: number;    // item difficulty (0-10)
    elapsed_days: number;  // days since last review
    scheduled_days: number; // scheduled interval
    reps: number;          // total repetitions
    lapses: number;        // times forgotten
    state: 0 | 1 | 2 | 3; // New=0, Learning=1, Review=2, Relearning=3
    last_review: number;   // timestamp
  };
}
```

**Grade Mapping:**
- Accuracy < 50% → Again (Rating.Again = 1)
- Accuracy 50-70% → Hard (Rating.Hard = 2)
- Accuracy 70-90% → Good (Rating.Good = 3)
- Accuracy > 90% → Easy (Rating.Easy = 4)
- Manual override: user can select rating after each review item

**Migration Strategy:**
- Existing records without `fsrsCard` → auto-initialize as Review state with `stability = nextReviewDelayDays` approximation
- `nextReview` field kept for backward compatibility, updated alongside `fsrsCard.due`

## Feature 3: Pronunciation Assessment

**Goal:** Add phoneme-level pronunciation scoring to the Speak module using SpeechSuper API (primary) with AI-based fallback.

### 3.1 Architecture: Dual Provider

```
User speaks → Record audio (MediaRecorder API)

         │  (phoneme scoring)     │

                    ↓ fallback

         │  + AI prompt analysis  │
         └────────────────────────┘
                    ↓
         Unified PronunciationResult
                    ↓
         Enhanced UI (phonemes, IPA, tips)
```

### 3.3 Unified Result Type

```typescript
interface PronunciationResult {
  provider: 'speechsuper' | 'ai';
  overallScore: number;        // 0-100
  fluencyScore: number;        // 0-100
  completenessScore: number;   // 0-100
  words: PronunciationWord[];
  tips: string[];              // improvement suggestions
}

interface PronunciationWord {
  word: string;
  score: number;               // 0-100
  phonemes?: PronunciationPhoneme[];
}

interface PronunciationPhoneme {
  phoneme: string;             // IPA symbol
  score: number;               // 0-100
  suggestion?: string;         // e.g., "Round your lips more"
}
```

### 3.4 SpeechSuper Integration

- API endpoint: `https://api.speechsuper.com/`
- Requires: `appKey` + `secretKey` (user configures in Settings)
- Sends audio (WAV/WebM) + reference text → returns phoneme-level scores
- Free tier: 1000 calls/month
- Quota tracking: store call count in localStorage, reset monthly

### 3.5 AI Fallback

When SpeechSuper quota is exceeded or not configured:
1. Use Web Speech API to get recognized text + confidence
2. Send to user's configured AI provider with prompt:
   - Reference text, recognized text, word-level comparison
   - Ask for: common pronunciation mistakes analysis, IPA hints, improvement tips
3. Parse AI response into `PronunciationResult` (scores estimated from text comparison + AI analysis)

### 3.6 UI Enhancements

After speaking, show enhanced feedback:
- **Overall Score Ring** — pronunciation score (replaces or supplements accuracy ring)
- **Word-level Phoneme Display** — each word expandable to show IPA phonemes with color-coded scores (green/yellow/red)
- **Pronunciation Tips** — actionable suggestions like "Focus on the 'th' sound in 'the' — place tongue between teeth"
- **Practice Mode** — tap a problem word to hear correct pronunciation and re-record just that word

### 4.4 Sync Engine Design

```typescript
// Sync strategy: last-write-wins based on updatedAt timestamp
interface SyncEngine {
  // Full sync: pull all remote → merge with local → push local changes
  fullSync(): Promise<SyncResult>;
  // Incremental: sync only changes since lastSyncedAt
  incrementalSync(): Promise<SyncResult>;
  // Push a single record immediately after local write
  pushRecord(table: string, record: any): Promise<void>;
}

// Sync flow:
// 1. Pull: fetch remote records where updated_at > lastSyncedAt
// 2. Merge: for each record, keep the one with later updatedAt
// 3. Push: upload local records where updatedAt > lastSyncedAt
// 4. Update lastSyncedAt timestamp
```

### 4.5 Dexie Hooks for Auto-Sync

Add hooks to Dexie tables that trigger sync after local writes:
- `db.contents.hook('creating')` → set `updatedAt`
- `db.contents.hook('updating')` → set `updatedAt`, queue for sync
- Same for `records` and `sessions`
- Debounced sync: batch local changes and push every 5 seconds

User clicks "Login with Google/GitHub"

## Implementation Order

The features have minimal dependencies on each other. Recommended order:

1. **FSRS** (Feature 1) — smallest scope, self-contained, immediate user value
2. **Analytics** (Feature 2) — independent, builds on existing data
3. **Pronunciation** (Feature 3) — independent, new capability
4. **Auth & Sync** (Feature 4) — largest scope, touches many files, do last

Features 1-3 can be developed in parallel as they don't share files.
